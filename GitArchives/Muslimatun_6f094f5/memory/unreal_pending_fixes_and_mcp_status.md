## unreal-mcp connection status (confirmed live as of 2026-07-01)
`unreal-mcp` now reliably exposes full toolsets (Actor, Asset, Blueprint, Material, MaterialInstance, StaticMesh, Texture, Scene, Object, Programmatic, etc.) whenever the Unreal Editor is open — confirmed working in a session that started *after* the Editor was already running. The earlier "never connects" problem was specific to sessions that started before the Editor launched. `editor_toolset.toolsets.programmatic.ProgrammaticToolset.execute_tool_script` is the key tool for batch work: it runs a sandboxed Python script (only `json/math/re/time/copy/datetime` + an `execute_tool(tool_name, json_input)` helper — no raw `unreal` module access) that can call any other registered MCP tool in a loop, avoiding hundreds of round trips. Always call `get_execution_environment` once first. Object refs for tools like `ObjectTools`/`MaterialInstanceTools` need the full object-path form `"/Game/Path/Package.AssetName"` (package path repeated as the asset name after the dot), not just the content path that `AssetTools.find_assets`/`get_referencers` return.

**3. Raw/undecimated mesh optimization — done, turned out to be mostly already mitigated.** Checked triangle count / LOD count / Nanite status on every heavy mesh named in the original audit (`Content/Level/Export/{Shape389,BlockV360,door039,3dSolid7716,Cube_053,Plane126,rack_012}`, `Statik_tree/Big_tree/{Ash-tree_05_21_2m_,18m_Terak_01,18m_Terak_02}`, `Statik_tree/leaf`, `Speed_tree/Sample_Broadleaf_Forest2/Nanite/N_Sample_Broadleaf_Forest2`, `Speed_tree/Cornus_sanguinea/*`, `BlankDefault/Model/{wicker,wicker1,zont_1_011}`, `GrandCapital/FBX/Models/Fountain/Pond`). All of them have 600K–2.7M triangles at LOD0 as suspected, **but almost all already had Nanite enabled** (which virtualizes/culls geometry per-pixel regardless of source triangle count, largely neutralizing the "renders millions of triangles always" problem even without traditional LODs). Only **`rack_012`, `wicker`, `wicker1`** had Nanite disabled with `lod_count=1` (i.e. genuinely un-mitigated) — enabled Nanite on all three via `StaticMeshTools.set_nanite_enabled` and saved. Did not generate traditional LODs on any of these: for the already-Nanite meshes it's largely redundant (Nanite supersedes discrete LODs for the primary render path), and auto-decimating raw, artifact-prone CAD geometry without visual review risks introducing new mesh errors.

## Round 3: full Details-panel actor/light/material audit via the existing Python script (same day)
User also wanted the original "full Content audit" (every actor/light/material's Details-panel values) redone now that the Editor is live — but discovered `unreal-mcp`'s `ProgrammaticToolset.execute_tool_script` sandbox **cannot** run this: it only allows composing calls to other registered MCP tools (no raw `unreal` module, no console-command execution tool exists in any exposed toolset). A live level has ~3400-4000 actors, ~1500 lights each with 60+ raw reflected properties — doing this per-actor over MCP round-trips would be tens of thousands of calls. The already-existing `Content/Python/audit_level_export.py` (see [[unreal-audit-python-script]]) solves this properly since it runs natively inside the Editor process with full `unreal` module access and a already-curated property scope — so the user ran it manually (Output Log > Python > `import audit_level_export; audit_level_export.run()`) for 3 of the 4 main levels:
- `Saved/AuditExport/level_audit_Glavniy_MirMIron_20260701_203039.json` (3440 actors, ~20MB)
- `Saved/AuditExport/level_audit_Pragulka_20260701_203131.json` (3973 actors, ~29MB)
- `Saved/AuditExport/level_audit_VR_20260701_202928.json` (3610 actors, ~25MB)
- `RenderMirMiron` was **not** exported — skip unless the user asks for it specifically.

**No Python/Node available in the shell** for analyzing these large JSON files (only a Windows Store `python3`/`python` stub — confirmed dead end again, see [[unreal-perf-texture-audit]]). Used PowerShell `ConvertFrom-Json` instead — parses a 20-29MB file in ~2 seconds, no problem. Wrote a summary script that groups actors by class, aggregates light intensity min/max/avg per light-component-class, counts mesh-without-material, and tallies mesh usage frequency — output saved to `Saved/AuditExport/summary_report_20260701.json` (small, safe to read directly) rather than ever loading the raw 20-29MB per-level files into the conversation directly.

**Key findings from the summary:**
- **Zero mesh components with no material** across all 3 levels — the "door039 has zero materials" finding from the original filesystem-only audit ([[unreal-perf-texture-audit]] §4) referred to the *raw imported asset's default slot*, not what's actually assigned once placed in these levels; materials are properly assigned at placement. Don't repeat that claim as if it means "materials are missing in the scene."
- **Every single PointLight (73-78 per level) has the exact same `Intensity=4`** — never individually tuned. Worth flagging to the user if they ever ask about lighting variety/realism.
- **SpotLight intensity differs a lot by level**: Glavniy_MirMIron max=100/avg≈18.6, but Pragulka and VR max≈222.26/avg≈41 — roughly 2x brighter on average. Plausibly compensating for Pragulka/VR having **no baked lighting** ([[unreal-project-levels]]) i.e. brighter dynamic lights to compensate for the lack of baked GI, but flag as a real inconsistency if the user wants visual parity across levels.
- RectLight (0.8-20, avg~0.9), DirectionalLight (7), SkyLight (1.6) are consistent across all 3 levels.
- The heavy raw CAD props from [[unreal-perf-texture-audit]] §4 (`door039`, `Shape389`, `BlockV360`, `3dSolid7716`, `Plane126`, `Cube_053`, `rack_012`) are each placed **only once per level** — confirms they're unique architectural elements, not mass-duplicated, so today's Nanite-enable fix on `rack_012` covers its only instance. `wicker` (Nanite-enabled today) is placed 14-28 times per level — a real multi-instance win from that fix.
- **Notable non-issue to know about:** Glavniy_MirMIron places `/Engine/BasicShapes/Cube` 522 times and `/Engine/EditorMeshes/Camera/SM_CineCam` 519 times — matches the 518 `BP_POI_C` (Point-of-Interest / hotspot) blueprint instances in that level. These are lightweight interactive-hotspot marker meshes (part of the ArchVizExplorer framework, see [[unreal-archvizexplorer-structure]]), not a hidden geometry-weight problem — don't flag this as a performance concern.
