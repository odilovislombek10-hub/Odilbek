2026-07-03. VR level (`/Game/Level/Odilbek/VR`) stutters/lags in headset. Root cause = the project's rendering config is tuned for max-quality FLAT (monitor) — Lumen + Hardware RT + TSR + Nanite@1.0 + ScreenPercentage 100 — which is brutal in stereo VR. Fix strategy: apply lighter settings ONLY in VR (so Glavniy/Pragulka keep their look), via VRPawn BeginPlay console commands + one project-wide-but-VR-only config flag.

**AUDIT — `Content/Python/vr_perf_audit.py`** (user runs in Editor Python console: `exec(open(r"D:/.../Content/Python/vr_perf_audit.py").read())`; writes `Saved/VR_Perf_Audit.txt` + logs `VRAUDIT:` lines). VR level results: 702 actors, 374 unique static meshes / 2452 draws, 1772 HISM foliage (190 hosts), **0 skeletal meshes** (no heavy characters — good), 3 DirectionalLights only (2 dynamic-shadow), 35 decals, 1 VolumetricCloud + UDS, ~30 detailed car BPs (Maybach/Range Rover/Kia/Gentra…), 3 Cesium actors but NO heavy Cesium3DTileset. **Biggest costs: 199 TRANSLUCENT material slots (car glass/windows) + 1392 two-sided slots + volumetric clouds + Lumen HWRT.** So the problem is the RENDER PATH, not actor count. NOTE: the script's triangle-count API returned 0 for every mesh (StaticMeshEditorSubsystem.get_number_triangles / EditorStaticMeshLibrary path failed on this build) — tri data missing; fix the tri fn if poly data is needed later.

**APPLIED (2026-07-03):**
- **VRPawn `EventBeginPlay`** (inside the existing `if IsHeadMountedDisplayEnabled` branch, via write_graph_dsl full-rewrite — `Development|ExecuteConsoleCommand` nodes): `r.Nanite.MaxPixelsPerEdge 2`, `r.VolumetricCloud.ReflectionRaySampleMaxCount 0`, `r.ScreenPercentage 90`. Compiled+saved.
  - **HWRT-OFF REMOVED per user (2026-07-03):** originally also set `r.Lumen.HardwareRayTracing 0` + `.Reflections.HardwareRayTracing 0`, but user said it lowers VR visual quality ("vizualizatsiyani tushirib yuboradi") — ArchViz reflections/glass/car-paint need HWRT. Removed both from BeginPlay AND their resets from EndPlay, so **HWRT stays fully ON in VR** (inherits config default True). Kept only the low-visual-impact wins (Nanite edge, cloud-reflection off, ScreenPct 90) + Instanced Stereo. Tradeoff: less perf headroom → recover it via CONTENT optimization (translucent/two-sided/cars) which does NOT touch visuals. **CONFIRMED write_graph_dsl full-rewrite of an event REPLACES the body (omitted nodes are removed, not left) — verified by readback; the input AddMappingContext chain stayed intact.** These run ONLY when an HMD is active (real VR), so flat levels are untouched. Reversible (edit the pawn). **IMPORTANT — cvars are GLOBAL engine state, not per-level, and PERSIST within a session** (a new PIE launch does NOT re-read ini; only editor restart / fresh packaged launch resets to ini defaults). So without a reset, playing VR then a flat level in the SAME editor session leaks HWRT=0 into the flat level. **FIX: added VRPawn `EventEndPlay`** that restores full quality on VR exit: `r.Lumen.HardwareRayTracing 1`, `r.Lumen.Reflections.HardwareRayTracing 1`, `r.Nanite.MaxPixelsPerEdge 1.0`, `r.VolumetricCloud.ReflectionRaySampleMaxCount 80`, `r.ScreenPercentage 100` (matches DefaultEngine.ini defaults). So flat levels always get full quality even in one continuous session. **VERIFY r.Lumen.HardwareRayTracing toggles at runtime** — if it's read-only on this build, HWRT stays on and we must move it to a VR device-profile/config route.
- **`Config/DefaultEngine.ini` `[/Script/Engine.RendererSettings]`**: added `vr.InstancedStereo=True` (both eyes one pass, ~30-40% render-thread CPU saving; only affects stereo, zero impact on flat). **Needs Editor restart + one-time shader recompile.**

**VR PLAY CRASH FIX (2026-07-03, ~17:11) — SUPERSEDES the InstancedStereo=True above:** User pressed VR Play in VR level → `EXCEPTION_ACCESS_VIOLATION reading 0x90` in `UnrealEditor_Renderer` (RenderThread; GameThread IsCrashed=false). Crash log (`Saved/Crashes/UECC-...1C72C413.../`) last render line = `LogRenderer: Recreating Persistent SBTs due to initializer changes: NumShaderSlotsPerGeometrySegment 1->2, NumGeometrySegments 0->2304` = Lumen **Hardware Ray Tracing** building the RT scene's persistent Shader Binding Table under **Instanced Stereo**. This is the well-documented **RT + Instanced Stereo VR crash** (Epic forums, UE5.6+; happens with RT shadows/Lumen HWRT + `vr.InstancedStereo` in stereo) — one of the two must be disabled. User chose (AskUserQuestion) to **KEEP HWRT** for visual quality → broke the pair on the Instanced-Stereo side: set **`vr.InstancedStereo=False`** + added **`r.RayTracing.NonBlockingPipelineCreation=0`** (forces blocking RT pipeline creation, avoids null RTPSO). HWRT stays fully ON; only cost is losing the ~30-40% stereo CPU saving (recover via CONTENT opt below). **Needs Editor restart** (vr.* + RT pipeline cvars read at startup, not PIE). If it STILL crashes after restart, the fallback is disabling HWRT for VR (the option user declined) — via VRPawn BeginPlay `r.Lumen.HardwareRayTracing 0`.
