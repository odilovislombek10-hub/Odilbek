## The bug (found + fixed 2026-07-02)
User noticed that identical chairs (same mesh `Object248`, same material `Wood_06` + `m_Fabric_basce_Inst2`) placed via one HISM component (`HISM_Object248`, actor `Actor_28` in `Glavniy_MirMIron`) rendered in two visibly different colorways (light tan wood/orange cushion vs. dark brown wood/maroon cushion) in a screenshot.

Two wrong theories were tried and ruled out before finding the real cause:
1. Baked-lighting/shadow position difference — ruled out because live query showed the HISM component's `Mobility` is already `Movable`, so it never participates in static lightmap baking at all.
2. The mesh's missing lightmap-UV-channel-1 bug (real, confirmed via `Saved/AuditExport/lightmap_uv_audit_*.json` — `Object248` has `LightMapCoordinateIndex=1` but only 1 UV channel exists, `will_be_black_when_baked=true`) — also ruled out for this specific case since it only matters for Static-mobility placements, and this HISM is Movable. (This UV bug is still real and affects 338 meshes project-wide per the audit; relevant if any *Static*-mobility placement of one of those meshes looks broken.)

**Actual root cause:** read the HISM component's `perInstanceSMData` (via `ObjectTools.get_properties` on the component, property name `perInstanceSMData` — an array of 4x4 transform matrices, one per instance) and computed the determinant of each instance's 3x3 rotation/scale submatrix. Exactly 12 of 24 instances had **determinant = -1** (mirrored/reflected transform) while the other 12 had **determinant = +1** (proper rotation). A mirrored transform flips the mesh's normals, which changes how it responds to specular/reflection lighting — same mesh, same material, but visibly different shading. Whoever set up this HISM (likely a CAD-import/instancing script) alternated chair orientation using a negative-scale mirror instead of a proper 180°-additional yaw rotation.

**First fix attempt (WRONG — corrected same session):** rewrote all 24 `perInstanceSMData` entries with the *same* rotation submatrix as the "normal" siblings (yaw 90°, +1-determinant), assuming the chair mesh was symmetric enough that this wouldn't change apparent facing. **This was wrong** — the user reported back (screenshot `image_2026-07-02_01-35-21.png`) that the fix made the previously-mirrored chairs face backward/away from the table. The mirrored instances weren't a random import glitch that happened to also rotate wrong — they were deliberately placed facing the *opposite* direction from their siblings (e.g. chairs facing each other across a table), and someone achieved that opposite-facing look via a mirror (negative determinant, breaks normals/shading) instead of a proper 180°-additional rotation (positive determinant, correct shading, same visual facing intent).

**Correct fix:** for the same 12 instances that had negative determinant, replaced their rotation with **yaw 90°+180°=270°** (`xPlane={x:0,y:-1,z:0}`, `yPlane={x:1,y:0,z:0}`, `zPlane={0,0,1}`, determinant **+1**) instead of copying the sibling's plain yaw-90° rotation. This preserves the original "facing the opposite way" intent while fixing the normals/shading, since it's a proper rotation rather than a reflection. Applied the equivalent fix to a second, independently-broken HISM component found the same way: `Actor_33` = `HISM_manutti_cross_cloth002` (the cushion/fabric mesh, a *separate* component from the wood frame `Object248` — the chair's frame and cushion are two different HISM actors sharing the same footprint) — its "normal" pattern was identity (`xPlane=(1,0,0)`, `yPlane=(0,1,0)`), so its mirrored instances got the 180°-proper-rotation equivalent: `xPlane={x:-1,y:0,z:0}`, `yPlane={x:0,y:-1,z:0}`, determinant +1. Checked a third nearby HISM (`Actor_34` = `HISM_metal_generic`, the metal legs) and found no mirroring there — all its instances were already proper 90°-step rotations (scale 1.2 uniform), so it needed no fix.

**Lesson:** when a mirrored (negative-determinant) instance transform is found, do NOT assume the fix is "make it match a same-looking sibling's rotation" — that only happens to be correct if the object is visually symmetric enough that facing direction doesn't matter. Instead, decompose the intent: the *position* is right, and the *reason* for the mirror was almost always "make this instance face a different direction than its sibling." The correct fix is the proper (positive-determinant) rotation that reproduces that same *apparent* facing (typically +180° yaw from the sibling's rotation, but confirm from the actual mirrored matrix's basis vectors — compare which axis flipped sign relative to the sibling to infer the intended facing offset), not just copying the sibling verbatim. Always get user visual confirmation after this class of fix before considering it done — the failure mode (backward-facing furniture) is not something determinant math alone catches.

Verified by recomputing determinants after each write (all instances +1) and saved via `AssetTools.save_assets` on the level path `/Game/Level/Odilbek/Glavniy_MirMIron` (note: `SceneTools.save_actor` fails with "not an external actor asset" for actors inside a non-OFPA level — must save the level asset itself instead). Also had to toggle `bVisible` false→true on each fixed HISM component after each `perInstanceSMData` rewrite to force the component's GPU instance buffer/cluster tree to rebuild — writing the property alone did not visibly update the already-rendered instances.

**Why:** the user's own hypothesis ("instance qilinganda muammo bo'ldi" — something went wrong during instancing) was correct and led directly to the right diagnostic (checking per-instance transform determinants) after two lighting-based theories were tried and disproven with live data.
**How to apply:** if any other HISM/ISM component shows the same "identical mesh+material but some instances look shaded differently" symptom, check `perInstanceSMData` determinants the same way (`ObjectTools.get_properties` → parse `xPlane`/`yPlane`/`zPlane` → 3x3 determinant) before assuming it's a lighting or material bug. A mix of +1 and -1 determinants within one component is the tell. Fix by rewriting the mirrored entries' rotation submatrix to match a same-determinant sibling instance, keeping translations untouched.

## Tooling notes from this session
- `EditorAppToolset.CaptureViewport` needs `captureTransform` and `annotations` passed explicitly even though the schema marks them optional/nullable — omitting them raises "needs a default value". Pass a zeroed `annotations` object (`gridSpacing/gridExtent/gridHeight/maxLabelDistance/maxLabels` all 0) to effectively disable the overlay.
- `CaptureViewport` produces heavily motion-blurred images whenever the `captureTransform` differs from the viewport's actual current camera position (the engine seems to treat the teleport as fast camera motion for TAA/motion-blur purposes) — even calling it twice in a row with the *same* transform did not clear the blur in this session. Prefer `CaptureAssetImage` (asset thumbnails, e.g. for materials) when possible, which does not have this issue. If a clean level viewport screenshot is truly needed, ask the user to move the camera manually first, or accept the blur and cross-verify important findings numerically instead (as done here via determinant math) rather than trusting a blurred render.
- `ObjectTools.set_properties` can write back a full struct-array property (like `perInstanceSMData`) in one call by passing the same JSON shape `get_properties` returns for it — useful for bulk per-instance transform edits without any dedicated "set instance transform" tool existing in `StaticMeshTools`.
