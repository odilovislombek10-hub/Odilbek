When repointing a plain (non-parameterized) `MaterialExpressionTextureSample` node's `Texture` property on a base `Material` via `ObjectTools.set_properties` + `MaterialTools.recompile`, child `MaterialInstanceConstant` assets keep showing the OLD texture as a referencer (`AssetTools.get_referencers`) even after: re-saving the MI, opening it via `OpenEditorForAsset`, and re-saving again. Tried on `/Game/Megascans/Surfaces/Concrete_Wall_uepobaiew/NewMaterial` and `NewMaterial1` (fixed to point `T_Concrete_Wall_uepobaiew_4K_{D,N}`/`T_ConcreteWall_uepobaiew_4K_ORDp` at the canonical root copies instead of the `NewFolder/` duplicates) — the base Materials themselves confirmed fixed (their own `get_expressions` now show the new Texture), but child MIs `M_Ma` and `NewFolder1/NewMaterial_Inst` still list the old `NewFolder/` textures as referencers, despite having zero exposed Texture-type parameters (`MaterialInstanceTools.list_parameters` shows only Scalar/Vector) — meaning this isn't a parameter override, it's UE5's `FMaterialCachedExpressionData` (a per-MaterialInterface cache of all textures used transitively, including inherited ones) not getting invalidated by an MCP-driven edit the way it would from the normal Material Editor "Apply" cascade.
**Why it matters:** don't assume `get_referencers` going to zero after a same-session direct-reflection material edit — for MIs, it may need a full Editor restart (forces `PostLoad` → fresh `UpdateCachedExpressionData`) before it actually clears. Do NOT delete an "orphaned" duplicate texture just because the base Material graph looks fixed — verify `get_referencers` is actually empty first, and if it isn't, don't force-delete.
**How to apply:** for future base-Material graph edits (not just MI parameter overrides) done via `ObjectTools.set_properties` on `MaterialExpressionTextureSample*` nodes, treat the fix as "source-of-truth graph corrected, but dependent MI referencer lists may lag until Editor restart" — tell the user this explicitly rather than claiming the duplicate is fully unreferenced. Affects the leftover Megascans `Concrete_Wall_uepobaiew/NewFolder/{T_Concrete_Wall_uepobaiew_4K_D,T_Concrete_Wall_uepobaiew_4K_N,T_ConcreteWall_uepobaiew_4K_ORDp}` — still on disk, not yet deleted, referenced only by `M_Ma`/`NewFolder1/NewMaterial_Inst`'s stale cache as of 2026-07-02. Also still unresolved: a 4th oddly-named lone texture `/Game/Megascans/Surfaces/Concrete_Wall_uepobaiew/1` used as `NewMaterial1`'s D/color slot — not yet compared against the canonical D texture, flagged but not investigated this session.
