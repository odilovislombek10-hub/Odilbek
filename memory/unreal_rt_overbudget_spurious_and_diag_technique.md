2026-07-03: Root-caused the persistent "RAY TRACING GEOMETRY - REQUESTED/ALWAYS-RESIDENT MEMORY OVER BUDGET (0 B / 15,999 EiB)" on-screen error with hard data.

**It is SPURIOUS — not a real memory problem.** memreport RayTracingGeometry stats (Glavniy_MirMIron): `Requested Memory = 0 MB`, `Referenced = 0 MB`, `Always Resident = 0 MB`, but `Resident Memory = 3001 MB`; actual `RayTracing Acceleration Structure Memory = 2286 MB` vs pool budget `ResidentGeometryMemoryPoolSizeInMB = 4096 MB`. So real RT geometry (2.3-3 GB) is UNDER the 4 GB budget — the "0 B" in the error = the reference-based counters reading zero, and "15,999 EiB" = 2^64 uninitialized/overflow budget. Cause: `r.RayTracing.UseReferenceBasedResidency` was overridden to **false** (engine Constructor default = **true**); in fixed-pool mode the reference-based over-budget checks fire from an uninitialized state. Big projects leave this cvar at default, so they never see the warning — matches user's "bigger projects don't have this." **Likely fix: restore `UseReferenceBasedResidency=1` (default) or remove the RT residency overrides entirely.** NOTE: this CORRECTS [[unreal_rt_geometry_budget_fix]] which claimed `UseReferenceBasedResidency=0` was the fix — the live data shows =0 leaves the broken accounting.

**Device profile does NOT differ editor vs PIE here (disproved earlier hypothesis):** live SearchCVars in editor AND during PIE both show `r.RayTracing.Shadows=0`, `Skylight=0`, `PoolSize=9216`. Editor uses `WindowsEditor` device profile (no CVar section), so the `[Windows DeviceProfile]` Shadows=True only applies to a packaged/standalone build, not editor or in-editor PIE. So "editor clean, Play shows error" is NOT a device-profile cvar flip. For VR level specifically: it uses a VR GameMode -> Play renders to the HMD/stereo, which is why an MCP flat-viewport capture looks clean even when the user sees the error in the headset.

**DIAGNOSTIC TECHNIQUE (this is how to get real numbers — MCP has no console-exec):** `Content/Python/capture_play_diag.py` runs in the Editor's Python console (`import capture_play_diag`) and executes console commands via `unreal.SystemLibrary.execute_console_command(get_editor_subsystem(UnrealEditorSubsystem).get_game_world(), cmd)` — including `rhi.DumpMemory` (RT AS / Texture2D / RenderTarget breakdown to log) and `memreport -full` (writes Saved/Profiling/MemReports/<map>-<ts>/*.memreport with a full sorted texture list: `MaxAllowedSize, Current/InMem, Format, LODGroup, Name, Streaming, ...`). Run it WHILE PIE is active (script prints PIE=True/False) to capture real Play state; run in editor for static state. `Streaming.ListStreamingTextures` is NOT a valid command in UE5.8 (use the memreport texture list instead). The .memreport line is too long for Read's chunking — slice with awk/sed by line range. This technique found [[unreal_planerovka_textures_real_vram_hog]] (97 floor-plan textures = 17.64 GB) and the spurious RT accounting above. MCP CaptureViewport base64 result is huge -> decode via PowerShell `[Convert]::FromBase64String` to scratchpad PNG, then Read.
