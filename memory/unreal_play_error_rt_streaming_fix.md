User pressed Play and got three on-screen errors: "TEXTURE STREAMING POOL OVER 32,867 MiB BUDGET", "RAY TRACING GEOMETRY - ALWAYS RESIDENT MEMORY EXCEEDS 20% OF THE BUDGET (0 B / 15,999 EiB)", "RAY TRACING GEOMETRY - REQUESTED MEMORY OVER BUDGET 2,174 GiB / 15,999 EiB". Diagnosed and fixed both same session, live via `unreal-mcp` (Editor was open).

## 1. Ray tracing memory explosion — root cause found in config
`Config/DefaultEngine.ini` deliberately disables `r.RayTracing.Shadows=False` and `r.RayTracing.Skylight=False` (comment: "Lumen HWRT GI/Reflections/Shadows hammasini qiladi; alohida RT o'chirildi (2x yukni oldi)"). But `Config/DefaultDeviceProfiles.ini`, `[Windows DeviceProfile]` section, had `CVars=r.RayTracing.Shadows=True` and `CVars=r.RayTracing.Skylight=True` — device-profile CVars apply at runtime (Play/Standalone) and win over DefaultEngine.ini, silently re-enabling both RT passes on top of Lumen HWRT. Combined with CesiumForUnreal streaming huge amounts of real-world geometry (which all needs ray-tracing acceleration structures/BLAS built), this produced the absurd "2,174 GiB requested / 15,999 EiB budget" numbers (the EiB figure is UE's overflow display when no real RT geometry memory budget cvar is configured).
**Fix:** changed both lines in `Config/DefaultDeviceProfiles.ini` (`[Windows DeviceProfile]`) to `False`, matching `DefaultEngine.ini`. Requires an Editor/game restart to take effect (device profile CVars apply at startup).

## 2. Texture streaming pool over by ~32GB — root cause found and fixed live via MCP
Cross-referenced the existing `Saved/AuditExport/mcp_texture_audit_20260701.json` (from [[unreal-perf-texture-audit]]/[[unreal-pending-fixes-and-mcp-status]]) against the live error: the 393 project-wide oversized (>2048px) textures, if stored uncompressed, total **~32.78 GB** — matching the reported "32,867 MiB" almost exactly. **99 of those 393** live in `Content/Level/Odilbek/Planerovka/` (and `Xonadonlar/1Block..8Block` subfolders) — floor-plan/apartment-layout scan images, some up to 9925×7017px (~70 megapixels), alone totaling ~18GB if uncompressed.
Verified live via `unreal-mcp` `ObjectTools.get_properties` on a sample (`/Game/Level/Odilbek/Planerovka/4blok_tipovoy-page-00001`): `CompressionSettings=TC_EditorIcon` (uncompressed BGRA8, 4 bytes/pixel, not BC7), `maxTextureSize=0` (native resolution, uncapped — this was intentional, see [[unreal-pending-fixes-and-mcp-status]] "Correction applied mid-run": capping these to 2048px would make the floor-plan text unreadable), and critically **`NeverStream=true`** — meaning all 99 of these textures are permanently, fully resident in VRAM regardless of whether the floor-plan viewer widget is even displaying them. That combination (uncompressed + native res + never-stream) is what actually blew the pool, not the resolution/compression choices themselves (which were correctly left alone for readability).
**Fix applied and verified via `execute_tool_script` (ProgrammaticToolset)**: set `NeverStream=false` on all 99 `Texture2D` assets under `/Game/Level/Odilbek/Planerovka` (found via `AssetTools.find_assets` scoped to that folder, `asset_type=/Script/Engine.Texture2D`), then `AssetTools.save_assets`. Result: `{"total_found": 99, "fixed": 99, "failed": [], "saved": true}` — 100% success, no failures. Verified post-fix on the sample texture: `NeverStream=false` confirmed via a follow-up `get_properties` call. Resolution and compression (TC_EditorIcon, maxTextureSize=0) were deliberately left untouched — this preserves crisp/readable text in the floor-plan viewer; only the never-stream flag was flipped so the normal texture streaming system now manages residency (loads full quality when the widget displays a plan, unloads when not needed) instead of forcing all 99 permanently resident.

**Why:** user hit real errors on Play and asked for root cause + fix, not just a generic performance pass.
**How to apply:** if streaming pool errors recur, re-run the same MCP-based cross-reference against `Saved/AuditExport/mcp_texture_audit_20260701.json` (regenerate via the Round 2 method in [[unreal-pending-fixes-and-mcp-status]] if it's gone stale) before assuming it's a new problem — check `NeverStream`/`CompressionSettings`/`maxTextureSize` on any newly-added oversized texture the same way. If RT errors recur after a config change, always diff `DefaultEngine.ini` against `DefaultDeviceProfiles.ini` for the same cvar under `[Windows DeviceProfile]` — device profiles silently win at runtime and are the most common source of "it's disabled in the ini but still happens in Play" bugs in this project.
